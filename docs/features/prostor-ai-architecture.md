# PROSTOR AI — Мультиагентная система для водоочистки

### Предиктивная система обслуживания на базе AI-агентов, увеличивающая retention и LTV клиентов
*(Система умного обслуживания на основе искусственного интеллекта: сама напоминает клиенту о замене фильтров, подбирает оборудование и увеличивает повторные продажи и пожизненную ценность каждого клиента)*

---

## Обзор

Мультиагентная AI-система (набор из 7 самостоятельных AI-модулей, каждый отвечает за свою задачу), надстроенная над существующим CRM PROSTOR (Aqua Kinetika). Замыкает полный цикл клиента: привлечение → диагностика → продажа → обслуживание → retention (возврат клиента за повторными покупками).

**Стек:** Python, LangGraph (оркестратор — «дирижёр», который управляет работой всех AI-агентов), FastAPI (API-слой — интерфейс для общения между компонентами системы), OpenAI API (LLM/Vision — языковая модель с возможностью анализа изображений), существующий NestJS бэкенд PROSTOR как data layer (слой доступа к данным).

**Ключевая ценность:** AI-powered CRM (CRM-система с искусственным интеллектом), которая закрывает весь customer journey (путь клиента от первого обращения до покупки и обслуживания) без участия менеджера для 80% типовых сценариев.

---

## Архитектура

### Точки входа (3 сценария)

| Сценарий | Вход | Первый агент | Описание |
|----------|------|-------------|----------|
| Холодный клиент | Фото проблемы (ржавый кран, мутная вода, накипь) | Discovery Agent | Не знает что нужно, нужна квалификация |
| Клиент с анализом | PDF/фото результата анализа воды | Water Analyst | Есть данные, нужна рекомендация |
| Существующий клиент | Фото установленного оборудования | Service Agent | Есть оборудование, нужно обслуживание/апгрейд |

Все три сценария сходятся в общую цепочку: подбор → расчёт → маршрут → коммуникация.

## Агенты

### 0. Discovery Agent (Агент первичной диагностики)

**Назначение:** Входная точка для холодного клиента (человек, который впервые обратился и не знает что ему нужно). Визуальная диагностика + диалоговый скрипт квалификации проблемы (серия вопросов, чтобы понять суть проблемы).

**Вход:**
- Фото проблемы (ржавый кран, мутная вода, накипь в чайнике, запах)
- Текстовое описание от клиента

**Логика работы:**
1. Vision LLM (языковая модель, умеющая «видеть» изображения) анализирует фото, определяет визуальные признаки (ржавчина → железо, белый налёт → жёсткость, мутность → механические примеси)
2. Генерирует уточняющие вопросы на основе увиденного:
   - Источник воды (скважина / колодец / центральное водоснабжение)
   - Давность проблемы
   - Наличие запаха
   - Количество проживающих
3. На основе ответов квалифицирует проблему и маршрутит:
   - Достаточно данных → Equipment Selector (прямой подбор)
   - Нужен точный анализ → рекомендация сдать воду + запись на выезд инженера

**Выход:** Структурированный объект `WaterProblemAssessment`

```typescript
interface WaterProblemAssessment {
  visualDiagnosis: {
    detectedIssues: ('iron' | 'hardness' | 'turbidity' | 'odor' | 'color')[];
    confidence: number; // 0-1
    imageAnalysis: string;
  };
  clientInfo: {
    waterSource: 'well' | 'borehole' | 'municipal';
    householdSize: number;
    problemDuration: string;
  };
  recommendation: 'direct_selection' | 'water_analysis_needed';
  estimatedParameters?: Partial<WaterAnalysisResult>;
}
```

**Инструменты:** OpenAI Vision API (gpt-4o)

### 1. Water Analyst Agent (Агент анализа воды)

**Назначение:** Парсинг (автоматическое извлечение данных) и интерпретация результатов анализа воды.

**Вход:**
- PDF или фото лабораторного анализа воды
- Или `estimatedParameters` от Discovery Agent

**Логика работы:**
1. Vision LLM извлекает параметры из документа (уже реализовано в water-analysis-parser)
2. Нормализует в стандартную структуру
3. Сравнивает с ПДК (СанПиН 1.2.3685-21)
4. Выделяет критические отклонения

**Выход:** Структурированный объект `WaterAnalysisResult`

```typescript
interface WaterAnalysisResult {
  parameters: {
    iron: number;        // мг/л, ПДК 0.3
    hardness: number;    // °Ж, ПДК 7.0
    pH: number;          // 6.0-9.0
    turbidity: number;   // ЕМФ, ПДК 2.6
    manganese: number;   // мг/л, ПДК 0.1
    hydrogen_sulfide: number; // мг/л, ПДК 0.003
    tds: number;         // мг/л
    chlorides: number;   // мг/л
    nitrates: number;    // мг/л
    bacteria: number;    // КОЕ/мл
  };
  violations: {
    parameter: string;
    value: number;
    limit: number;
    severity: 'warning' | 'critical';
  }[];
  source: 'lab_report' | 'visual_estimate';
  confidence: number;
}
```

**Инструменты:** OpenAI Vision API, water-analysis-parser (существующий модуль)

**Статус:** Базовая версия реализована (water-analysis-parser)

### 2. Equipment Selector Agent (Агент подбора оборудования)

**Назначение:** Экспертный подбор оборудования на основе параметров воды. Работает как опытный инженер — по результатам анализа определяет какие фильтры нужны.

**Вход:**
- `WaterAnalysisResult` от Water Analyst
- Информация о клиенте (количество человек, бюджет, тип жилья)

**Логика работы (экспертные правила):**

```
ЕСЛИ iron > 0.3 мг/л:
  → Обезжелезиватель (каталитический или аэрационный)
  ЕСЛИ iron > 3.0: аэрационная колонна
  ЕСЛИ iron 0.3-3.0: картриджный обезжелезиватель

ЕСЛИ hardness > 7.0 °Ж:
  → Умягчитель (ионообменный)
  ЕСЛИ hardness > 15: кабинетный умягчитель
  ЕСЛИ hardness 7-15: картриджный умягчитель

ЕСЛИ turbidity > 2.6 ЕМФ:
  → Механический фильтр (предфильтр)

ЕСЛИ bacteria > 0:
  → УФ-стерилизатор

ЕСЛИ tds > 500:
  → Система обратного осмоса

ВСЕГДА:
  → Финишный фильтр (угольный) для вкуса и запаха

РАСЧЁТ ПРОИЗВОДИТЕЛЬНОСТИ:
  потребление = кол-во_человек × 200 л/сутки
  пиковый_расход = кол-во_человек × 0.5 л/мин
```

2. По определённой конфигурации ищет конкретные модели в каталоге МойСклад
3. Подбирает совместимые компоненты (фитинги, трубки, крепёж)
4. Формирует комплект с основным и альтернативным вариантом

**Выход:** `EquipmentAndServiceRecommendation`

```typescript
interface EquipmentAndServiceRecommendation {
  primaryConfig: {
    items: {
      moyskladId: string;
      name: string;
      category: 'filter' | 'cartridge' | 'fitting' | 'accessory';
      purpose: string;
      quantity: number;
    }[];
    reasoning: string;
  };
  alternativeConfig?: typeof primaryConfig;
  services: {
    type: 'installation' | 'maintenance' | 'diagnostics' | 'cartridge_replacement' | 'system_flush' | 'water_sampling';
    name: string;
    description: string;
    estimatedDuration: number; // минуты
    requiredEquipment?: string[]; // что инженер должен взять с собой
    price?: number;
  }[];
  cartridgeSchedule: {
    item: string;
    replacementIntervalDays: number;
    factors: string[];
  }[];
  serviceSchedule: {
    service: string;
    intervalMonths: number;
    description: string;
  }[];
  installationRequired: boolean;
}
```

**Подбор услуг (главный источник дохода):**

```
ВСЕГДА при новом оборудовании:
  → Услуга: Доставка и установка
  → Услуга: Пусконаладка и проверка
  → Услуга: Инструктаж клиента

ЕСЛИ система обратного осмоса или колонна:
  → Услуга: Ежегодное ТО (промывка, проверка давления)
  → Услуга: Замена мембраны (раз в 1.5-2 года)

ЕСЛИ картриджная система:
  → Услуга: Замена картриджей (по графику Predictive Reminders)

ЕСЛИ проблемная вода (iron > 3.0 или hardness > 15):
  → Услуга: Контрольный анализ воды через 2 недели после установки
  → Услуга: Повторный анализ через 6 месяцев

ВСЕГДА:
  → Услуга: Регулярное обслуживание (план на год)
  → Предложение: Договор на сервисное обслуживание (подписка)
```

**Инструменты:** MoySklad API (через mcp-moysklad), LLM для reasoning

### 3. Price Calculator Agent (Агент расчёта стоимости)

**Назначение:** Автоматический расчёт стоимости оборудования И услуг. Формирует готовое коммерческое предложение с полной картиной: сколько стоит купить + установить + обслуживать.

**Вход:**
- `EquipmentAndServiceRecommendation` от Equipment Selector

**Логика работы:**
1. Проверяет наличие каждой позиции на складе через МойСклад API
2. Получает актуальные цены на оборудование
3. Рассчитывает стоимость услуг (установка, пусконаладка, обслуживание)
4. Применяет скидки (программа лояльности, объём заказа, пакет «оборудование + услуги»)
5. Считает годовую стоимость обслуживания (замена картриджей + выезды инженера)
6. Формирует коммерческое предложение с разбивкой: разовые затраты vs регулярные

**Выход:** `PriceQuote`

```typescript
interface PriceQuote {
  equipment: {
    name: string;
    price: number;
    quantity: number;
    inStock: boolean;
    deliveryDays?: number;
  }[];
  services: {
    name: string;
    type: 'one_time' | 'recurring';
    price: number;
    recurringIntervalMonths?: number;
  }[];
  summary: {
    equipmentTotal: number;
    oneTimeServicesTotal: number;    // установка, пусконаладка
    totalUpfront: number;            // разовые затраты (оборудование + установка)
    monthlyServiceCost: number;      // ежемесячная стоимость обслуживания
    yearlyServiceCost: number;       // годовая стоимость обслуживания
    discount: number;
    grandTotal: number;              // всё вместе за первый год
  };
  subscriptionOffer?: {
    monthlyPrice: number;            // цена подписки на обслуживание
    includes: string[];              // что входит
    savings: number;                 // экономия по сравнению с разовыми вызовами
  };
  validUntil: string; // ISO date
}
```

**Ключевой момент:** Предложение подписки на обслуживание (subscriptionOffer) — это и есть основа монетизации. Клиент платит фиксированную сумму в месяц, а компания получает стабильный recurring revenue (регулярный доход).

**Инструменты:** MoySklad API (остатки, цены), PROSTOR API (скидки, лояльность, прайс на услуги)

### 4. Service Agent (Агент обслуживания)

**Назначение:** Обслуживание существующего оборудования. По фотографии распознаёт что установлено у клиента, оценивает состояние, рекомендует замену картриджей или апгрейд (переход на более современную модель).

**Вход:**
- Фото установленного оборудования
- История клиента (если есть в CRM)

**Логика работы:**
1. Vision LLM распознаёт модели фильтров по фото (Аквафор Морион, Кристалл, Трио и т.д.)
2. Определяет визуальное состояние (загрязнение картриджей, протечки, повреждения)
3. По модели определяет совместимые картриджи
4. По истории клиента + количеству человек рассчитывает следующую замену
5. Рекомендует апгрейд если система устарела или не справляется

**Выход:** `ServiceRecommendation`

```typescript
interface ServiceRecommendation {
  equipment: {
    model: string;
    condition: 'good' | 'needs_maintenance' | 'needs_replacement';
    installedDate?: string;
  };
  cartridges: {
    type: string;
    currentState: string;
    nextReplacementDate: string;
    moyskladId: string;
  }[];
  upgradeRecommendation?: {
    reason: string;
    suggestedModel: string;
  };
  action: 'cartridge_replacement' | 'maintenance' | 'upgrade' | 'ok';
}
```

**Инструменты:** OpenAI Vision API, PROSTOR API (история клиента), MoySklad API (картриджи)

### 5. Predictive Reminders Module (Модуль предиктивных напоминаний — «умные» напоминания о замене)

**Назначение:** Предиктивные напоминания о замене картриджей — система сама предсказывает когда у конкретного клиента закончится ресурс фильтра и заранее напоминает. (Killer feature — уникальная функция, которой нет у конкурентов на рынке.)

**Текущая реализация:** Расчёт по количеству человек в доме. Бот Telegram напоминает о замене.

**AI-расширение:**

```
БАЗОВЫЙ РЕСУРС КАРТРИДЖА = паспортный_ресурс_литры

КОРРЕКТИРУЮЩИЕ КОЭФФИЦИЕНТЫ:
  k_household = кол-во_человек / 3  // нормировка к стандарту
  k_water_quality = параметр / ПДК   // чем хуже вода, тем быстрее износ
  k_season = 1.0 (лето) | 0.85 (зима) // летом потребление выше
  k_history = avg(факт_замены) / avg(расчёт_замены) // корректировка по истории

РЕАЛЬНЫЙ РЕСУРС = БАЗОВЫЙ × (1 / k_household) × (1 / k_water_quality) × k_season × k_history

ДАТА ЗАМЕНЫ = дата_установки + (РЕАЛЬНЫЙ РЕСУРС / (кол-во_человек × 200)) дней
```

**Факторы:**
- Количество человек → потребление воды
- Параметры анализа воды → скорость износа картриджей
- Сезонность → летом потребление выше
- История замен конкретного клиента → индивидуальная корректировка
- Тип картриджа → разный базовый ресурс

**Каналы уведомлений:**
- Web push — за 7 дней до расчётной даты замены
- Email — детальное письмо с ценами и кнопкой заказа
- В приложении — баннер при входе

### 6. Route Planner Agent (Агент планирования маршрутов)

**Назначение:** Оптимизация маршрутов инженеров. Решает задачу коммивояжёра (как объехать все точки за минимальное время) с учётом приоритетов и загрузки.

**Вход:**
- Список заявок на выезд (установка, замена, диагностика)
- Локации клиентов
- Доступные инженеры и их текущие позиции

**Логика работы:**
1. Получает список заявок из PROSTOR API
2. Группирует по географии (кластеризация)
3. Решает задачу коммивояжёра через OSRM API (уже реализовано в PROSTOR)
4. AI-оптимизация: учёт приоритетов (срочные заявки), окон доступности клиентов, загрузки инженеров

**Выход:** `RouteOptimization`

```typescript
interface RouteOptimization {
  routes: {
    engineerId: string;
    stops: {
      clientId: string;
      address: string;
      lat: number;
      lng: number;
      arrivalTime: string;
      taskType: 'installation' | 'maintenance' | 'diagnostics';
      estimatedDuration: number; // минуты
    }[];
    totalDistance: number; // км
    totalTime: number; // минуты
  }[];
  unassigned: string[]; // заявки без слота
}
```

**Инструменты:** OSRM API (существующая интеграция), PROSTOR API (заявки, инженеры)

**Статус:** Базовая маршрутизация реализована в PROSTOR. AI-оптимизация — расширение.

### 7. Client Communicator Agent (Агент коммуникации с клиентом)

**Назначение:** Мультиканальная коммуникация — общение с клиентом через разные каналы (чат на сайте, email, push-уведомления) с единой логикой.

**Архитектура (Adapter Pattern — паттерн адаптера, позволяет добавлять новые каналы без переписывания логики):**

```
[Web Chat] ←→ [Web Adapter (polling/WS)] ←→ [Unified Message Bus] ←→ [Agent Core] ←→ [Orchestrator]
[Email]    ←→ [Email Adapter (SMTP)]     ←→ [Unified Message Bus]
[Web Push] ←→ [Push Adapter (SW)]        ←→ [Unified Message Bus]
```

**Каналы:**
1. **Web Chat** — основной канал, polling (MVP) → WebSocket (v2 для стриминга LLM)
2. **Email** — документы, результаты анализа, коммерческие предложения, напоминания
3. **Web Push** — оперативные уведомления (инженер выехал, заказ готов)

**Adapter Pattern:**
- Каждый канал имеет свой адаптер
- Адаптер нормализует входящее сообщение в единый формат
- Response Router адаптирует ответ под канал:
  - Web Chat → Rich UI (карточки товаров, кнопки, изображения)
  - Email → HTML шаблон с брендингом
  - Web Push → короткий текст + deep link в приложение

**Unified Message Format:**

```typescript
interface UnifiedMessage {
  userId: string;
  channel: 'web' | 'email' | 'push';
  messageType: 'text' | 'image' | 'document' | 'action';
  payload: string;
  attachments?: { type: string; url: string }[];
  metadata: {
    sessionId: string;
    timestamp: string;
    intent?: string;
  };
}
```

**Intent Detection:**
- Простые вопросы (FAQ, статус заказа) → прямой ответ без оркестратора
- Фото проблемы → Discovery Agent
- Фото анализа → Water Analyst
- Фото оборудования → Service Agent
- Запрос на подбор → Equipment Selector

## Оркестратор (LangGraph) — «Дирижёр» системы

**Назначение:** Управление потоком между агентами. Определяет какой агент должен работать следующим, передаёт данные между ними, обрабатывает ошибки. Как дирижёр в оркестре — сам не играет, но управляет кто когда вступает.

**State Graph (граф состояний — схема переходов между агентами):**

```
START → intent_detection
  → "cold_client"      → discovery → water_analyst? → equipment_selector → price_calculator → route_planner → communicator → END
  → "has_analysis"     → water_analyst → equipment_selector → price_calculator → route_planner → communicator → END
  → "existing_client"  → service_agent → price_calculator → route_planner → communicator → END
  → "simple_query"     → communicator → END
```

**Shared State (общее состояние — единая «память» в которую каждый агент записывает свой результат, и следующий агент его читает):**

```typescript
interface ProstorAIState {
  messages: Message[];
  clientId: string;
  entryPoint: 'cold' | 'analysis' | 'existing' | 'simple';
  waterAssessment?: WaterProblemAssessment;
  analysisResult?: WaterAnalysisResult;
  recommendation?: EquipmentRecommendation;
  priceQuote?: PriceQuote;
  serviceInfo?: ServiceRecommendation;
  route?: RouteOptimization;
  currentAgent: string;
  error?: string;
}
```

## Бизнес-модель: оборудование — вход, услуги — основной доход

### Экспертная оценка бизнеса водоочистки

В сфере водоочистки, как и в большинстве сервисных бизнесов, прослеживается типичная закономерность:

**Товары — точка входа клиента.** Основной объём продаж приходится на оборудование (фильтры, умягчители, системы обратного осмоса). Однако маржинальность товаров имеет тенденцию к снижению — закупочные цены растут быстрее розничных, а номинальный рост выручки часто совпадает с инфляцией.

**Услуги — основной источник прибыли.** Маржинальность услуг (установка, обслуживание, замена картриджей) как правило в 1.5-2 раза выше маржинальности товаров. Себестоимость услуги — это в основном труд инженера, она не зависит от закупочных цен на оборудование и не подвержена инфляционному давлению.

**Типичные проблемы сервисного направления:**
- Нехватка квалифицированных инженеров — найти и удержать специалистов сложно
- При ограниченном штате инженеров каждый час на вес золота — нужна максимальная автоматизация
- Клиенты не знают какая услуга им нужна — из десятков видов услуг сложно выбрать
- Клиенты забывают про замену картриджей — уходят к конкурентам или используют просроченные фильтры
- Менеджеры перегружены ручной обработкой заявок

**Потенциал роста:** каждый проданный фильтр = повторный заказ на обслуживание через 6-12 месяцев. Если системно предлагать услуги каждому клиенту (а не ждать пока сам попросит), доля сервисной выручки может вырасти кратно.

### Как AI-агенты решают эти проблемы

| Проблема | Решение AI-агентов | Результат |
|----------|-------------------|-----------|
| Клиент не знает какая услуга нужна | Discovery Agent квалифицирует по фото за секунды | Клиент получает конкретную рекомендацию, а не блуждает по каталогу из 100+ услуг |
| Клиенты забывают про замену картриджей | Predictive Reminders напоминает заранее с учётом качества воды и расхода | Повторные продажи растут без затрат на маркетинг |
| Инженер тратит время на дорогу | Route Planner оптимизирует маршруты (OSRM уже работает) | Больше выездов в день при том же штате |
| Менеджер вручную обрабатывает каждый заказ | Client Communicator закрывает 80% обращений без менеджера | Менеджеры занимаются только сложными кейсами |
| Нет приоритизации услуг | Equipment Selector + Price Calculator подбирают и оценивают под ситуацию | Система приоритизирует маржинальные услуги (коттеджный сегмент, умягчители) |
| Нет системного cross-sell | Equipment Selector предлагает услуги вместе с оборудованием | Конверсия «товар → товар + услуга» растёт |

### Воронка монетизации (как AI генерирует доход)

| Этап | Что делает AI | Доход |
|------|--------------|-------|
| Привлечение | Discovery Agent квалифицирует клиента по фото | — |
| Диагностика | Water Analyst парсит анализ воды | Услуга: выезд + забор воды |
| Подбор | Equipment Selector рекомендует оборудование + услуги | Продажа оборудования |
| Установка | Route Planner планирует выезд инженера | **Услуга: доставка + установка + пусконаладка** |
| Обслуживание | Predictive Reminders напоминает о замене | **Услуга: замена картриджей (регулярно)** |
| ТО | Service Agent определяет состояние по фото | **Услуга: ежегодное ТО** |
| Подписка | Price Calculator предлагает сервисный договор | **Recurring revenue (постоянный доход)** |
| Апгрейд | Service Agent рекомендует модернизацию | Повторная продажа оборудования + установка |

### Целевые метрики

| Метрика | Ожидаемое текущее | Цель с AI (6 мес) | Простыми словами |
|---------|-------------------|-------------------|------------------|
| Доля услуг в продажах | ~5-10% | 15-20% | Каждый 5-й рубль — за услугу |
| Конверсия товар → товар + услуга | Низкая | 15% | Каждый 7-й покупатель берёт ещё и установку |
| Шагов до заказа услуги | 6-8 | 2-3 | Клиент заказывает за 2 тапа, а не блуждает по меню |
| Время ответа клиенту | Часы (рабочее время) | Секунды | AI отвечает мгновенно 24/7 |
| Повторные заказы (замена) | Только Telegram | Telegram + web push + email | Клиент получает напоминание там где удобно |
| Сервисные контракты | Нет | 10-15% клиентской базы | Стабильный предсказуемый доход каждый месяц |

## Почему это реально, а не фантастика

Данная архитектура — не проект «с нуля». Это надстройка над существующей production-системой PROSTOR, которая работает в боевом режиме более 13 месяцев. Ключевые компоненты уже реализованы и протестированы на реальных данных:

| Компонент | Статус | Что уже работает |
|-----------|--------|-----------------|
| NestJS бэкенд | ✅ В проде, 59 модулей | Auth, заказы, корзина, чат, уведомления, платежи |
| OpenAI Vision интеграция | ✅ В проде | Парсинг анализов воды по фото (water-analysis-parser) |
| МойСклад API | ✅ В проде | Каталог оборудования, остатки, цены, синхронизация |
| OSRM маршрутизация | ✅ В проде | Оптимизация маршрутов инженеров, задача коммивояжёра |
| Бот напоминаний | ✅ В проде | Напоминания о замене картриджей по количеству человек |
| Платежи ЮКасса | ✅ В проде | Онлайн-оплата заказов |
| PostgreSQL + PostGIS | ✅ В проде | Хранение данных, гео-запросы |
| Redis | ✅ В проде | Кеширование, сессии, очереди |
| MCP-сервер МойСклад | ✅ Open source | Готовый коннектор для AI-агентов к складу |

**Что нужно добавить:** оркестрационный слой на Python (LangGraph) — «дирижёр», который свяжет существующие компоненты в единую AI-систему. Это не переписывание, а надстройка. Существующий NestJS бэкенд продолжает работать как data layer, новый Python-сервис обращается к нему через API.

**Аналогия:** представьте дом, в котором уже есть фундамент, стены, крыша, электричество и водопровод. Мы не строим новый дом — мы устанавливаем систему «умный дом», которая управляет всем что уже построено.

## Глоссарий (словарь терминов)

| Термин | Что это простыми словами |
|--------|------------------------|
| **AI-агент** | Самостоятельный AI-модуль, который умеет выполнять конкретную задачу (например, распознавать фото или подбирать фильтры) |
| **Мультиагентная система** | Несколько AI-агентов, работающих вместе и передающих друг другу данные по цепочке |
| **Оркестратор (LangGraph)** | «Дирижёр» — программа, которая решает какой агент должен работать следующим и передаёт данные между ними |
| **Vision LLM** | Языковая модель, которая умеет не только читать текст, но и «видеть» изображения (распознавать что на фото) |
| **LLM (Large Language Model)** | Большая языковая модель — AI типа ChatGPT, который понимает и генерирует текст |
| **Retention** | Удержание клиента — когда клиент возвращается к нам снова, а не уходит к конкурентам |
| **LTV (Lifetime Value)** | Пожизненная ценность клиента — сколько денег клиент принесёт за всё время сотрудничества |
| **Killer feature** | Уникальная функция продукта, которой нет у конкурентов и которая даёт решающее преимущество |
| **MVP (Minimum Viable Product)** | Минимально жизнеспособный продукт — первая рабочая версия с базовыми функциями |
| **Adapter Pattern** | Паттерн проектирования «адаптер» — позволяет подключать новые каналы (email, push) без переписывания основной логики |
| **Strangler Fig Pattern** | Паттерн постепенной миграции — новая система обвивает старую и постепенно заменяет её (как фикус-душитель обвивает дерево) |
| **Event-driven** | Событийная архитектура — компоненты общаются через события («произошла оплата» → «отправить уведомление»), а не напрямую |
| **Message Bus (шина сообщений)** | Общий канал, через который все части системы обмениваются сообщениями |
| **Polling** | Способ получения данных: клиент периодически спрашивает сервер «есть что новое?» |
| **WebSocket** | Постоянное соединение между клиентом и сервером — данные приходят мгновенно в обе стороны |
| **API (Application Programming Interface)** | Интерфейс для общения программ между собой — как «меню ресторана» для программ |
| **Intent detection** | Определение намерения — AI понимает чего хочет клиент по его сообщению |
| **Deep link** | Ссылка, которая открывает конкретный экран внутри приложения, а не главную страницу |
| **ПДК** | Предельно допустимая концентрация — максимально допустимое содержание вещества в воде по нормативам |
| **СанПиН** | Санитарные правила и нормы — государственные стандарты качества воды |
| **FastAPI** | Современный фреймворк на Python для создания серверных приложений |
| **LangGraph / LangChain** | Фреймворки на Python для создания AI-приложений и мультиагентных систем |
