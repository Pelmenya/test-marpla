# Уровень 2 — Memory (память и диалоги)

## Зачем Memory

LLM **stateless** — каждый запрос обрабатывается с нуля. Memory = Flowise хранит историю и подкладывает её в каждый запрос как контекст.

## Типы Memory

| Тип | Как работает | Плюсы | Минусы | Когда использовать |
|-----|-------------|-------|--------|-------------------|
| **Buffer Memory** | Все сообщения в RAM | Полный контекст | 💰 Дорого, ⚠️ упирается в лимит токенов, теряется при рестарте | Короткие сессии (5-10 сообщений) |
| **Buffer Window Memory** | Последние N в RAM | Контроль стоимости и лимита | Теряется при рестарте, "забывает" старое | MVP без persistence |
| **Conversation Summary Memory** | LLM сжимает историю в саммари | Компактно даже на 1000+ сообщений | 💰 Доп. вызов LLM на каждое сообщение, теряет детали | Многочасовые разговоры |
| **Upstash/Redis/Zep Memory** | Persistent хранилище | Переживает рестарт, multi-instance | Нужна доп. инфраструктура | Production |

## Рабочая связка для production

```
Buffer Window Memory (N=10) + Redis/Upstash
```

- **Window** — контролирует **что** идёт в LLM (экономия токенов)
- **Redis** — контролирует **где хранится** (persistence, multi-instance)

## Как работает под капотом

```json
POST https://api.openai.com/v1/chat/completions
{
  "messages": [
    { "role": "system", "content": "Ты support-бот" },
    { "role": "user", "content": "Привет, я Дима" },
    { "role": "assistant", "content": "Здравствуйте, Дима!" },
    { "role": "user", "content": "Какой у меня заказ?" },
    { "role": "assistant", "content": "Ваш заказ №123" },
    { "role": "user", "content": "Новое сообщение" }
  ]
}
```

**Каждый запрос отправляет всю историю целиком.** LLM ничего не помнит между вызовами.

## 💰 Стоимость растёт линейно

| Сообщение | Токены в запросе |
|-----------|-----------------|
| 1-е | 200 |
| 5-е | 1000 (200 × 5) |
| 10-е | 2000 (200 × 10) |
| 50-е | 10 000 — упрёмся в лимит модели |

**Вывод:** Window Memory — не просто удобство, это экономия денег.

## sessionId — мультипользовательский режим

| Откуда взять | Паттерн | Когда использовать |
|-------------|---------|-------------------|
| Явно с фронта | `"sessionId": "..."` в overrideConfig | Простые интеграции |
| Из JWT | `sessionId = user.id` | Support-бот, один диалог на пользователя |
| `userId + threadId` | `"user-42-thread-1"` | ChatGPT-style с несколькими диалогами |
| UUID в cookie | `anonymous-uuid-xxx` | Неавторизованные пользователи |

### Пример передачи

```json
POST /api/v1/prediction/{chatflowId}
{
  "question": "...",
  "overrideConfig": {
    "sessionId": "user-42"
  }
}
```

## ⚠️ Подводные камни Memory

| Проблема | Решение |
|----------|---------|
| Утечка данных между пользователями | sessionId **только** из доверенного источника (JWT), не из body |
| История растёт бесконечно | TTL в Redis + endpoint "сбросить разговор" |
| Multi-tenant смешивается | `sessionId = tenantId:userId` |
| Перезапуск = амнезия | Persistent Memory (Redis/Upstash) |
| LLM забывает старое | Conversation Summary Memory для длинных диалогов |
| Token limit exceeded | Window Memory с разумным N (10-20) |

## Типовые кейсы

### Кейс 1 — Simple chatbot

```
Buffer Window Memory (N=10) ──┐
Prompt Template ──────────────┼──► LLM Chain
ChatOpenAI ───────────────────┘
```

**Plain RAM, теряется при рестарте, для MVP.**

### Кейс 2 — Production support-бот

```

Prompt Template ────────────────────────────┼──► LLM Chain
ChatOpenAI ─────────────────────────────────┘
```

**Persistence + контроль токенов.**

```
Conversation Summary Memory + Redis ──┐
Prompt Template ──────────────────────┼──► LLM Chain
ChatOpenAI ───────────────────────────┘
```

**Саммари держит суть, Redis — persistence. 💰 Дорого, но работает.**

## Вопросы для самопроверки

1. Почему нельзя "один раз сказать LLM контекст, а потом только новое сообщение слать"?
   > LLM stateless. Каждый API-вызов — чистый лист. Контекст нужен в каждом запросе.

2. Зачем Window Memory если есть Buffer Memory?
   > Buffer растёт бесконечно → упирается в token limit модели (8k/16k/128k) и дорожает. Window ограничивает последними N.

3. Когда Summary Memory оправдан?
   > Когда диалог > 50 сообщений и каждое важно помнить хоть в общих чертах. На коротких диалогах overhead не окупается.

4. Как защититься от смешивания сессий разных пользователей?
   > sessionId строго из JWT на бэкенде, не принимать от клиента напрямую. Префикс `tenantId` в multi-tenant.

## Чек-лист Memory в production

- [ ] Memory подключена к LLM Chain (не забыта)
- [ ] Тип Memory соответствует сценарию (Window для MVP, Redis для prod)
- [ ] sessionId передаётся из доверенного источника (JWT)
- [ ] Есть endpoint/механизм сброса сессии
- [ ] TTL на историю в Redis (например, 30 дней)
- [ ] Максимальная длина истории ограничена (Window N)
- [ ] Логируется использование токенов (для анализа затрат)
