# Уровень 3 — RAG (Retrieval Augmented Generation)

## Что такое RAG

**RAG = архитектурный паттерн "Retrieval + Generation".**

LLM не знает твоих данных. RAG даёт ей возможность "подсматривать" в документы перед ответом:

```
Пользователь: "Какая гарантия на фильтр Аквафор?"
       ↓
1. Retrieval — ищем релевантные куски в базе
2. Augmentation — добавляем найденное в промпт
3. Generation — LLM отвечает на основе контекста
```

⚠️ **RAG ≠ embeddings.** RAG — это подход. Retrieval может быть любым: full-text search, SQL WHERE, векторный поиск, гибрид.

---

## Методы поиска (Retrieval)

| Метод | Как работает | Плюсы | Минусы | Когда использовать |
|-------|-------------|-------|--------|-------------------|
| **Full-text (tsvector)** | Встроен в PostgreSQL, ищет по словам/леммам | Бесплатно, быстро, GIN-индекс | Не понимает синонимы | Точные ключевые слова |
| **pg_trgm (триграммы)** | Поиск по подстрокам | Ищет с опечатками | Для коротких строк | Автокомплит, имена |
| **Vector search (pgvector)** | Поиск по смыслу через embeddings | Синонимы, парафразы | 💰 Нужны embeddings | Расплывчатые запросы |
| **Hybrid** | Full-text + vector с ранжированием | Лучшее качество | Сложнее | Production-уровень |

## Важно: tsvector ≠ pgvector

Два разных расширения PostgreSQL:

| tsvector | pgvector |
|----------|----------|
| Встроен в PostgreSQL | Отдельное расширение |
| Поиск по словам | Поиск по векторам (embeddings) |
| `to_tsvector()`, `@@` | `VECTOR(1536)`, `<=>` |
| Бесплатно | Нужна embedding-модель |

## Embeddings — что это и как работают

**Embedding** = вектор чисел, описывающий смысл текста. Похожие тексты → близкие векторы.

```
"фильтр для жёсткой воды" → [0.12, -0.45, 0.87, ...]
"умягчитель"              → [0.14, -0.43, 0.89, ...]  ← близко
"холодильник"             → [-0.67, 0.12, -0.11, ...] ← далеко
```

### Косинусное расстояние

Мера близости векторов (корреляция Пирсона для центрированных векторов):
- **0.0** = одинаковый смысл (параллельные)
- **1.0** = ортогональные (несвязанные)
- **2.0** = противоположные

Оператор в pgvector: `embedding <=> query_vector`

### Как модель "знает" синонимы

Обучалась на миллиардах текстов, выучила что "умягчитель" и "фильтр от жёсткой воды" встречаются в похожих контекстах → дала им близкие векторы. Никто не писал правила.

## Embedding-модели

| Модель | Провайдер | Размерность | Цена | Когда |
|--------|-----------|-------------|------|-------|
| **text-embedding-3-small** | OpenAI | 1536 | $0.02/1M | Production по умолчанию |
| **text-embedding-3-large** | OpenAI | 3072 | $0.13/1M | Критична точность |
| **bge-m3** | BAAI (локально) | 1024 | $0 | 🔥 Лучший open-source, RU+EN |
| **nomic-embed-text** | Nomic (локально) | 768 | $0 | Быстрая, лёгкая |
| **mxbai-embed-large** | MixedBread (локально) | 1024 | $0 | Обгоняет OpenAI small в бенчмарках |

### Локально vs OpenAI

**Локально (Ollama + bge-m3 на GPU 16GB):**
- ✅ Бесплатно
- ✅ Приватность
- ✅ Batch-индексация миллионов записей
- ❌ Требует GPU
- ❌ Не масштабируется на сотни RPS пиком

**OpenAI:**
- ✅ Infinity scale, параллельные запросы
- ✅ Удобство (API-вызов)
- ❌ Стоит денег
- ❌ Данные уходят наружу

### Стратегия для проектов

- **Dev / пет-проекты** → локально через Ollama
- **Prod real-time** (пользовательский поток) → OpenAI
- **Prod batch** (разово индексируем миллионы) → локально + перенос векторов

## Pipeline RAG в Flowise

```
┌─── INDEXING (один раз при загрузке документа) ───┐
│                                                   │
│  Document Loader → Text Splitter → Embeddings → │
│  (PDF/CSV/URL)    (chunks)        (→ векторы)    │
│                                          ↓        │
│                                    Vector Store   │
│                                                   │
└───────────────────────────────────────────────────┘

┌─── QUERY (каждый запрос) ────────────────────────┐
│                                                   │
│  Question → Embeddings → Vector Store (top-K) → │
│                          релевантные chunks →    │
│                          LLM + промпт → Ответ   │
│                                                   │
└───────────────────────────────────────────────────┘
```

### Ноды в Flowise

| Категория | Нода | Что делает |
|-----------|------|-----------|
| **Document Loaders** | PDF, CSV, URL, Text, GitHub, Notion | Загрузка документов |
| **Text Splitters** | Recursive Character, Token, Markdown | Разбиение на chunks |
| **Embeddings** | OpenAI, Ollama, HuggingFace, Cohere | Текст → вектор |
| **Vector Stores** | pgvector, Pinecone, Qdrant, Chroma | Хранение и поиск |
| **Retrievers** | Similarity, MMR, MultiQuery | Стратегии извлечения |
| **Chains** | Conversational Retrieval QA | Готовый RAG-pipeline |

## Chunks (разбиение документа)

### Размеры и когда какой

| Chunk Size | Плюсы | Минусы | Когда |
|------------|-------|--------|-------|
| **100 токенов** | Точный поиск | Потеря контекста | Редко |
| **500 токенов** | 🔥 Баланс | — | По умолчанию |
| **1000 токенов** | Больше контекста | Размытость embedding | Нарративные тексты |
| **2000+** | Почти документ целиком | Embedding "размазан" | Не рекомендуется |

### Overlap (перекрытие)

Стандарт: **chunk 500 + overlap 50 (10%)**.

Зачем overlap — мысль не разорвётся пополам между чанками, контекст сохраняется.

```
Без overlap:
  Chunk 1: "Гарантия на картриджи 2 года."
  Chunk 2: "Условия включают температуру..."  ← о чём? непонятно

С overlap 50:
  Chunk 1: "Гарантия на картриджи 2 года."
  Chunk 2: "...картриджи 2 года. Условия..."  ← контекст сохранён
```

### Типы сплиттеров в Flowise

| Сплиттер | Когда |
|----------|-------|
| **Recursive Character Text Splitter** | 🔥 По умолчанию, разбивает от \n\n → \n → ". " → " " |
| **Character Text Splitter** | Тупое по одному символу |
| **Token Text Splitter** | Точность в токенах, дороже |
| **Markdown Text Splitter** | Разбивает по заголовкам |
| **Code Text Splitter** | Уважает границы функций/классов |
| **HTML Text Splitter** | Парсит веб-страницы |

⚠️ **Подвох Flowise:** Chunk Size по умолчанию в **символах**, не токенах. 1 токен ≈ 4 символа. Для 500 токенов ставь ~2000 символов.

### Таблица решений

| Документ | Chunk | Overlap | Почему |
|----------|-------|---------|--------|
| FAQ | 200-400 | 0 | Каждый Q&A атомарный |
| Регламенты | 500 | 50 | Стандарт |
| Длинные статьи | 800-1000 | 100 | Нарратив |
| Код | Семантически | 0 | По функциям |
| Чат-логи | По сообщениям | 0 | Атомарные |
| Каталог товаров | 1 товар = 1 chunk | 0 | Независимые |

### Семантическое разбиение

Более умный способ — разбивать **по смыслу**, а не по размеру:
- Считаем embedding каждого предложения
- Соседние предложения с резко разным смыслом → граница chunk

**Плюсы:** идеальные границы  
**Минусы:** 💰 дорого, нет в Flowise из коробки

## pgvector — индексы и поиск

### Создание таблицы

```sql
CREATE EXTENSION vector;

CREATE TABLE chunks (
    id SERIAL PRIMARY KEY,
    document_id INT,
    content TEXT,
    embedding VECTOR(1536)  -- размерность зависит от модели
);
```

### Типы индексов

| Индекс | Как работает | Плюсы | Минусы |
|--------|-------------|-------|--------|
| **Без индекса** | Full scan | Точный 100% | ⚠️ O(N), медленно на больших объёмах |
| **IVFFlat** | Кластеры + поиск в ближайших | Быстрее full scan | Нужно пересобирать |
| **HNSW** | 🔥 Граф связей между векторами | Быстрый, ~95-99% точность, не требует пересборки | Больше памяти |

**Для production — HNSW:**

```sql
CREATE INDEX ON chunks USING hnsw (embedding vector_cosine_ops);
```

### Поиск

```sql
-- Топ-5 ближайших
SELECT content, embedding <=> $query_vector AS distance
FROM chunks
ORDER BY embedding <=> $query_vector
LIMIT 5;
```

Операторы:
- `<=>` — косинусное расстояние (обычно используется)
- `<->` — L2 (евклидово)
- `<#>` — inner product

## Стратегии Retrieval

### Similarity Search (по умолчанию)

Просто top-K ближайших векторов. Быстро, но может выдать дубликаты.

### MMR (Maximum Marginal Relevance)

Балансирует **релевантность и разнообразие** — не берёт похожие чанки, даёт разные.

```
Запрос: "фильтры от жёсткой воды"

Similarity:
  1. Картриджный умягчитель A
  2. Картриджный умягчитель B  ← похож на A
  3. Картриджный умягчитель C  ← тоже похож

MMR:
  1. Картриджный умягчитель
  2. Ионообменная колонна      ← другой тип
  3. Обратный осмос            ← третий подход
```

### MultiQuery Retriever

Генерирует несколько вариантов запроса через LLM, ищет по каждому, объединяет результаты.

```
Пользователь: "что от накипи?"

LLM генерирует варианты:
  - "фильтр от жёсткой воды"
  - "умягчитель воды"
  - "система от известкового налёта"

Поиск по всем трём → объединение → топ-K
```

🎯 Лучше качество, но 💰 дороже (4-5 запросов к LLM на один вопрос).

## Когда RAG НЕ нужен

- **Генерация с нуля** (SEO-описание товара) — контекст не нужен
- **База маленькая** (< 100 записей) — положи в промпт целиком
- **Простой поиск по ключевым словам** — tsvector справится без LLM
- **Нет смысловых запросов** — пользователи всегда пишут точно → достаточно SQL

## Типовые кейсы

### Кейс 1 — FAQ-бот

```
PDF FAQ → Recursive Splitter (200, overlap 0) → OpenAI Embeddings 
                                                      ↓
                                                   pgvector
                                                      ↓
User question → Embeddings → Similarity Search (top-3) → LLM → Ответ
```

### Кейс 2 — Каталог товаров (рекомендации)

```
Товары из БД → 1 товар = 1 chunk → OpenAI Embeddings → pgvector

User: "что-то от ржавой воды" → Embedding → Top-5 товаров → LLM с контекстом
```

### Кейс 3 — Поиск по регламентам компании

```
Markdown документы → Markdown Splitter (500, overlap 50) → bge-m3 (локально) → pgvector

Question → bge-m3 → MMR Retriever (top-5) → Llama 3.1 8B (локально) → Ответ
```

**Полностью локальный RAG — бесплатно после разовой настройки.**

### Кейс 4 — Гибрид для e-commerce

```
Full-text search (tsvector) → топ-50 по ключевым словам
      +
Vector search (pgvector)    → топ-50 по смыслу
      ↓
Reciprocal Rank Fusion → финальные топ-10 → LLM
```

## Чек-лист RAG в production

- [ ] Выбран подходящий метод retrieval (full-text / vector / hybrid)
- [ ] Chunk size и overlap соответствуют типу документа
- [ ] Embedding-модель выбрана под бюджет и качество
- [ ] Vector store с HNSW-индексом на production нагрузках
- [ ] Есть стратегия переиндексации (модель меняется — векторы тоже)
- [ ] Абстракция над embedding-провайдером (возможность переключить dev↔prod)
- [ ] Сохраняется оригинальный текст (не только векторы)
- [ ] Есть мониторинг качества retrieval (находятся ли нужные chunks)
- [ ] Top-K настроен под размер контекстного окна LLM

## ⚠️ Подводные камни

| Проблема | Решение |
|----------|---------|
| Векторы от разных моделей несовместимы | Пиши абстракцию, меняй модель — переиндексируй всё |
| Chunk режется посреди мысли | Добавь overlap 10% |
| Низкое качество поиска | Попробуй hybrid (text + vector) или MMR |
| LLM отвечает не по контексту | В промпте: "Отвечай ТОЛЬКО на основе контекста, если не знаешь — скажи" |
| Дорого индексировать | Начни с локальных embeddings (Ollama) |
| Нельзя скрыть данные | Локальные модели, не отправляй в OpenAI |
